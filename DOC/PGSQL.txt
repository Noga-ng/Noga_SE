C'est une excellente question. Cette table est souvent celle qui pose le plus de difficultés au début.

Le problème à résoudre

Imaginons deux tables.

Étudiants
students

id | nom
---+---------
1  | Alice
2  | Bob
3  | Noga
Cours
courses

id | nom
---+-------------
1  | PostgreSQL
2  | PHP
3  | Docker

Maintenant, réponds à cette question :

Quels cours suit Alice ?

Alice peut suivre :

PostgreSQL
PHP
Docker

Ensuite :

Qui suit le cours PostgreSQL ?

Alice
Bob
Noga

On a donc :

un étudiant peut suivre plusieurs cours
un cours peut être suivi par plusieurs étudiants

C'est une relation plusieurs-à-plusieurs (N → N).

Pourquoi ne pas mettre course_id dans students ?

Par exemple :

students

id | nom   | course_id
---+-------+----------
1  | Alice | 1

Mais Alice suit aussi PHP.

Tu fais quoi ?

id | nom   | course_id
1  | Alice | 1
1  | Alice | 2

Tu répètes Alice plusieurs fois. C'est une mauvaise conception.

Ou alors :

course_id = 1,2,3

C'est du texte, pas une vraie relation. Impossible à exploiter correctement avec SQL.

Pourquoi ne pas mettre student_id dans courses ?

Même problème.

courses

id | nom        | student_id
---+------------+-----------
1  | PostgreSQL | 1

Mais PostgreSQL est suivi par Bob aussi.

Tu serais obligé de dupliquer le cours.

La solution : une table de liaison

On crée une troisième table.

CREATE TABLE student_course (
    student_id INTEGER REFERENCES students(id),
    course_id INTEGER REFERENCES courses(id),
    PRIMARY KEY(student_id, course_id)
);

Cette table ne contient que les associations.

Exemple :

student_course

student_id | course_id
-----------+----------
1          | 1
1          | 2
1          | 3
2          | 1
3          | 1
3          | 3

Cela signifie :

Alice suit PostgreSQL
Alice suit PHP
Alice suit Docker
Bob suit PostgreSQL
Noga suit PostgreSQL
Noga suit Docker
Pourquoi une clé primaire composée ?
PRIMARY KEY(student_id, course_id)

Cela signifie que la combinaison (student_id, course_id) doit être unique.

Par exemple :

student_id | course_id
-----------+----------
1          | 2

est autorisé.

Mais ajouter une deuxième fois :

student_id | course_id
-----------+----------
1          | 2

sera refusé.

En revanche :

1 | 1
1 | 2
1 | 3

est parfaitement valide.

La clé primaire empêche simplement les doublons d'inscription au même cours.

Visualisation
students                student_course              courses

1 Alice      ------\
                     \                 /------ 1 PostgreSQL
2 Bob         -------+--------------/
                     |
3 Noga       --------+--------------\------ 2 PHP
                     |
                     \---------------------- 3 Docker

La table student_course joue le rôle d'un pont entre students et courses.

Dans la vie réelle

Ce modèle est utilisé partout :

Utilisateur ↔ Rôle (user_roles)
Produit ↔ Catégorie (product_categories)
Film ↔ Acteur (movie_actors)
Livre ↔ Auteur (book_authors)
Employé ↔ Projet (employee_projects)
Utilisateur ↔ Permission (user_permissions)

Dès que chaque élément peut être lié à plusieurs éléments de l'autre table, on utilise une table de liaison.

C'est l'une des structures les plus importantes à maîtriser en conception de bases de données relationnelles.

| Contrainte  | Rôle                                  |
| ----------- | ------------------------------------- |
| PRIMARY KEY | Identifie une ligne de manière unique |
| FOREIGN KEY | Relie deux tables                     |
| NOT NULL    | Interdit les valeurs `NULL`           |
| UNIQUE      | Empêche les doublons                  |
| DEFAULT     | Donne une valeur par défaut           |
| CHECK       | Vérifie une condition                 |
